## Individual Reflection (Sisi)

In our activity, I saw a clear difference between sequential and parallel execution in how work is done. The sequential version processed one flow of recursive merge steps in a single process, so behavior was simple to track and debug. The parallel version split the list into chunks, sent each chunk to worker processes, and then merged chunk results back together. That structure gave us more moving parts, so we had to think about process startup, chunk boundaries, and final merge order before thinking about raw speed.

The timing behavior changed a lot as dataset size increased. In my benchmark run, sequential was faster for small input sizes: at 1,000 elements it averaged 0.001232s while parallel averaged 0.045731s, and at 10,000 elements sequential averaged 0.017061s while parallel averaged 0.037470s. At larger sizes, the result flipped. At 100,000 elements, sequential averaged 0.210641s and parallel averaged 0.105682s. At 1,000,000 elements, sequential averaged 2.754691s and parallel averaged 1.041813s. This matched what we discussed in class: larger workloads give parallel workers enough work to offset setup cost.

I ran into practical implementation issues while working with multiprocessing. The main one was making sure code runs inside `if __name__ == "__main__":` when starting processes, because running from stdin caused process startup errors. I also had to watch chunking logic, since chunk size comes from integer division and can create uneven final chunks. Another concern was correctness during merge passes, because each round has to preserve order and include every element exactly once.

The biggest insight for me was that parallelism adds overhead before it adds speed. Creating a `Pool`, moving data to workers, waiting for workers to finish, and merging sorted chunks all cost time. Synchronization points happen each time the program waits for mapped tasks and each time merge rounds reduce the list of sorted chunks. If those costs are high compared to the work size, parallel loses. If the array is large enough, the extra coordination cost becomes smaller than the time saved by running chunk sorts at the same time.

Parallelism was beneficial for our medium to large sorting workloads, especially at 100,000 and 1,000,000 elements where it produced close to 2x to 2.6x speedup in my run. Parallelism was unnecessary for small inputs like 1,000 and 10,000 elements, where sequential finished sooner and with less complexity. Based on this activity, I would choose sequential for quick small tasks and use parallel only when the workload is large enough to pay for process and merge overhead.
