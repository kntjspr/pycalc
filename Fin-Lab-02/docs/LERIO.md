# Individual Reflection – Distributed Voting System (Fin-Lab-02)

Working on this distributed voting system using Supabase helped me understand how distributed systems actually behave outside of theory. Compared to our previous activities that focused more on local processes and threads, this lab introduced a different environment where components communicate through networks and failures can happen anytime.

While implementing the edge node, I realized that even simple vote generation had an important role in the whole system. In edge_node.py, the use of uuid.uuid4() allowed each generated vote to act like it came from a different user device. Adding retry logic also changed how I viewed failures in programming. In normal sequential programs, an error usually stops execution immediately, but in distributed systems the node is expected to keep retrying and continue operating even if communication temporarily fails. That difference became more obvious during testing.

The Cloud Run API layer also showed why distributed systems separate responsibilities between services. The API only validates the incoming vote and forwards it to Pub/Sub without doing heavy processing. At first, it seemed unnecessary to add another messaging layer instead of writing directly to Firestore, but during the worker failure test I understood its purpose better. Even after disabling the worker service, the API still accepted votes normally while Pub/Sub stored the pending messages. Firestore stopped updating temporarily, but the entire system did not crash. The failure only affected one part of the architecture instead of everything.

The worker service helped me understand idempotency in a more practical way. Since the document ID was built using both user_id and poll_id, duplicate messages did not create duplicate records in Firestore. During fault injection, when duplicate votes were intentionally sent multiple times, Firestore still produced the correct final result because existing documents were simply overwritten instead of duplicated. This prevented incorrect vote counts even with repeated deliveries from Pub/Sub.

One thing I noticed during recovery testing was the cold-start delay in Cloud Run. After restoring the worker service, there was a short waiting time before queued messages started processing again because the container still needed to start up. This showed me one trade-off of serverless systems: they are scalable and cost-efficient, but recovery is not always instant.

The most important thing I learned from this activity was how distributed systems can continue functioning even when some components fail. Pub/Sub acted like a buffer between services, allowing messages to stay safe while the worker was offline. Once the worker came back, the queued votes were processed automatically without needing manual recovery. Watching the system recover by itself helped me better understand eventual consistency, where different parts of the system may temporarily have different states but eventually synchronize correctly over time.

Overall, this activity gave me a better understanding of fault tolerance, asynchronous communication, and distributed architecture. It also showed me that designing distributed systems is not only about writing code, but also about organizing services in a way that allows the system to remain reliable even during failures.
